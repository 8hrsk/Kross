# Настройка и запуск Сервера Kross

Сервер Kross (`cmd/kross-server`) отвечает за онлайн-активацию лицензий, предотвращение несанкционированного использования одного ключа на нескольких устройствах (особенно для массовых одноразовых ключей) и ведение глобального черного списка отозванных ключей.

## Сборка

Сборка сервера без внешних C-зависимостей (pure Go SQLite):
```bash
go build -o dist/kross-server ./cmd/kross-server
```

---

## Запуск сервера

Запуск с параметрами по умолчанию:
```bash
./dist/kross-server
```

### Настройки флагов:
- `-addr` — адрес и порт, который будет прослушивать HTTP-сервер (по умолчанию `:8080`).
- `-db` — путь к файлу базы данных SQLite (по умолчанию `kross.db`).

Пример запуска на нестандартном порту с базой в `/var/lib/kross/`:
```bash
./dist/kross-server -addr="0.0.0.0:9000" -db="/var/lib/kross/licensing.db"
```

*Примечание: База данных создается автоматически при первом запуске вместе со всеми необходимыми таблицами и индексами.*

---

## Архитектура базы данных (SQLite)

Сервер использует СУБД SQLite (реализована на чистом Go через драйвер `modernc.org/sqlite`, что устраняет необходимость компиляции с GCC / CGO).

### Схема таблиц:
*   `activations` — логирует все успешные активации ключей на устройствах.
    *   `license_id` (TEXT) — уникальный UUID лицензии.
    *   `license_type` (TEXT) — тип лицензии (`personal` или `mass`).
    *   `email` (TEXT) — email, указанный при активации.
    *   `hwid_hash` (TEXT) — хэш-идентификатор железа (HWID).
    *   `activated_at` (DATETIME) — дата активации.
    *   `revoked` (BOOLEAN) — статус блокировки данной записи.

Уникальный индекс установлен на связку `(license_id, hwid_hash)`. Это позволяет одному и тому же ПК повторно активировать один и тот же ключ без создания дубликатов в БД и без ложных срабатываний защиты.

---

## Логика валидации и защиты от пиратства

При отправке запроса на `/api/v1/activate` сервер обрабатывает его по следующим правилам в зависимости от типа лицензии:

### 1. Персональная лицензия (`personal`)
- Разрешено регистрировать один и тот же ключ на **нескольких различных HWID**. Привязка идет к email адресу. Это позволяет легитимному пользователю использовать свой софт на домашнем ПК и ноутбуке.

### 2. Одноразовая массовая лицензия (`mass`)
- При первой отправке запроса с нового HWID создается запись активации.
- При повторной активации с **тем же HWID** — сервер возвращает статус успешной активации (это считается повторным запуском или переустановкой приложения на том же ПК).
- При попытке активации этого же ключа на **другом HWID** (другом ПК):
  1. Сервер помечает все предыдущие активации этого ключа как `revoked` (отозванные).
  2. Ключ заносится в черный список.
  3. Сервер возвращает ответ со списком заблокированных ключей, чтобы клиентские приложения при синхронизации немедленно деактивировали данный софт.

## API Эндпоинты

### 1. `POST /api/v1/activate`
Вызывается клиентом для регистрации активации.

**Тело запроса (JSON):**
```json
{
  "license_id": "c1a8bc4a-9b4e-4f30-8d59-3fb9a3d46a81",
  "license_type": "mass",
  "email": "user@domain.com",
  "hwid_hash": "e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855"
}
```

**Ответ в случае успеха (200 OK):**
```json
{
  "status": "activated"
}
```

**Ответ при обнаружении пиратства (200 OK, ключ заблокирован):**
```json
{
  "status": "revoked",
  "revoke_keys": ["c1a8bc4a-9b4e-4f30-8d59-3fb9a3d46a81"],
  "message": "License key revoked due to multiple hardware activations"
}
```

---

### 2. `GET /api/v1/revocations`
Используется клиентами для периодической проверки отозванных лицензий.

**Ответ (200 OK):**
```json
{
  "revoked_keys": [
    "c1a8bc4a-9b4e-4f30-8d59-3fb9a3d46a81",
    "f81d4fae-7dec-11d0-a765-00a0c91e6bf6"
  ]
}
```
Клиент получает этот список, сохраняет его локально в зашифрованное хранилище и сверяет с ним текущую лицензию при старте приложения.
